# Task 2: Robust cleanup with retry, startup check, and fallback

## Description

This task implements three-layer resilience for the cleanup sequence in geo-fix (R-4 fix from analyst review).

**Problem:** Currently `cleanup()` in `system_config.py` already collects a `failures` list, but on failure there is no retry, no persistence, and the user notification in `_do_cleanup()` only prints to stderr. If cleanup partially fails (e.g., certutil is busy, registry locked by another process), the system is left in a dirty state with no automatic recovery path. The user has to remember to run `geo-fix --cleanup` manually.

**Solution — 3 layers (in order):**

1. **Retry with delay:** Each cleanup step that fails is retried once after a 3-second pause. This covers transient locks (certutil busy mid-write, registry locked by an installer).

2. **User notification:** If retry also fails, immediately notify the user what remains unclean — stderr print plus tray toast (using the existing `GeoFixTray.notify()` API). The user is informed but does not need to act.

3. **Startup cleanup:** On next launch, geo-fix reads `cleanup_pending.json` from the app data directory and re-executes any failed operations before starting a new session. This is the real safety net — the user just launches geo-fix again, which they do naturally.

**State persistence:** A `cleanup_pending.json` file in the app data dir records failed operations as a JSON list of string labels. It is deleted after successful startup cleanup.

**Scope:** `system_config.py` owns the retry logic, pending-file write/read, and `check_pending_cleanup()`. `main.py`'s `_do_cleanup()` owns the tray toast notification and the startup call to `check_pending_cleanup()`.

## What to do

1. **Refactor `cleanup()` in `src/system_config.py`** to wrap each of the 5 cleanup steps (CA cert removal, session tmpdir deletion, proxy restore, Firefox restore, firewall removal) in individual try/except blocks. On failure, wait 3 seconds and retry that step once. Collect labels of steps that still fail after retry into a `list[str]`. Return this list from `cleanup()` (currently returns implicitly — make the return explicit).

2. **Add `CLEANUP_PENDING_FILE` constant** pointing to `Path(os.environ.get("APPDATA", Path.home())) / "geo-fix" / "cleanup_pending.json"` in `system_config.py`.

3. **Add `write_cleanup_pending(failed_ops: list[str]) -> None`** in `system_config.py` — creates the app data dir if needed and writes the list as JSON.

4. **Add `check_pending_cleanup() -> None`** in `system_config.py` — reads `cleanup_pending.json` if it exists, executes each pending operation by label (match labels used in step 1), deletes the file on complete success, logs a warning if any re-execution fails.

5. **Update `_do_cleanup()` in `src/main.py`** — after collecting failures from `cleanup()`, if failures remain: write `cleanup_pending.json` via `write_cleanup_pending()`, and show a tray toast notification listing what couldn't be cleaned (in addition to the existing stderr print). Access the `tray` instance through a module-level reference or closure.

6. **Call `check_pending_cleanup()` in `main()` in `src/main.py`** — before `create_session_tmpdir()`, after any stale state cleanup, so leftover artifacts from the previous session are removed before the new session starts.

7. **Update imports in `main.py`** — import `write_cleanup_pending`, `check_pending_cleanup` from `src.system_config`.

## Details

**Files:**

- `src/system_config.py` — currently has `cleanup()` at line 653 that already collects a `failures` list but has no retry logic. The function currently does `return failures` at line 720 inconsistently (no return type annotation). The 5 steps are: CA cert removal, session tmpdir deletion, proxy restore, Firefox restore, firewall removal. Each step label must be a stable string constant (e.g., `"CA cert removal"`, `"Session tmpdir deletion"`, `"Proxy restore"`, `"Firefox restore"`, `"Firewall removal"`) — these same labels are used to dispatch re-execution in `check_pending_cleanup()`.

- `src/main.py` — `_do_cleanup()` at line 227 calls `cleanup(state)` and already prints to stderr on failure. Need to add: call `write_cleanup_pending(failures)` before the stderr print, and call `tray.notify(message)` to show a toast. The `tray` instance is not currently accessible inside `_do_cleanup()` — add a module-level `_tray = None` reference and assign it when the tray is created in `main()`. Call `check_pending_cleanup()` in `main()` around line 486, after stale state cleanup but before `create_session_tmpdir()`.

**Dependencies:** Task 4 (firewall cleanup by prefix) must be complete first — `remove_firewall_rules()` is called inside `cleanup()` and its new implementation is the one being wrapped with retry.

**Edge cases:**
- `check_pending_cleanup()` called when `cleanup_pending.json` does not exist — must be a no-op
- `check_pending_cleanup()` cannot find the matching operation for a stored label — log warning, skip that label, still delete file if all others succeed
- `write_cleanup_pending()` fails to write (permissions, disk full) — log warning, do not raise, do not mask the original cleanup result
- Tray not initialized when `_do_cleanup()` runs (e.g., crash before tray started) — guard the `_tray.notify()` call with `if _tray is not None`
- `cleanup_pending.json` app data dir: use `Path(os.environ.get("APPDATA", Path.home())) / "geo-fix"` — consistent with any existing app data usage in the project
- The 3-second retry delay is a real `time.sleep(3)` — tests must mock `time.sleep` to avoid slow tests

**Implementation hints:**
- Use the same step-label strings for both writing to JSON and dispatching in `check_pending_cleanup()`. Define them as module-level constants or as a list generated from step definitions.
- `check_pending_cleanup()` should call the same underlying functions as `cleanup()` (e.g., `uninstall_ca_cert(thumbprint=None)` for CA cert label) using stateless/best-effort variants — it has no state object at startup.
- The `stateless_cleanup()` function already does stateless artifact removal; `check_pending_cleanup()` can call individual sub-functions rather than the full `stateless_cleanup()` to target only what failed.
- Tests should use `tmp_path` (pytest fixture) for the pending file location — patch `CLEANUP_PENDING_FILE` constant or pass it as a parameter.
