# User Spec: resource-optimization

## Что делаем

Оптимизация потребления RAM и CPU приложения geo-fix. Устранение неограниченного роста памяти при долгой работе (mitmproxy flow retention, WebSocket message accumulation), сокращение базового потребления RAM за счёт минимизации загружаемых аддонов mitmproxy, и снижение CPU-нагрузки при активном browsing.

## Зачем

geo-fix работает как фоновая утилита весь рабочий день (8-18 часов). Текущее потребление 200-300MB RAM с ростом до 400-500MB неприемлемо: пользователь работает с браузером (30+ вкладок), IDE и Docker параллельно. Рост памяти со временем — главная проблема: через несколько часов приложение занимает непропорционально много ресурсов для фоновой утилиты.

## Как должно работать

1. Пользователь запускает geo-fix как обычно (без изменений в CLI/UI)
2. Приложение стартует с потреблением ~100-150MB RAM
3. При активном использовании (15+ вкладок Google, Google Docs, Meet) RAM остаётся стабильной
4. Через 8 часов непрерывной работы потребление не превышает 150MB
5. CPU при обычном серфинге не превышает 5% в среднем
6. Все текущие функции работают без изменений: spoofing, tray icon, watchdog, country switching, setup wizard, cleanup

## Критерии приёмки

**Ресурсы (измеряются как Private Working Set в Task Manager / PowerShell):**
- [ ] Базовое потребление RAM при старте не превышает 150MB (основной процесс + watchdog)
- [ ] RAM стабильна после 8 часов работы — Private Working Set не растёт более чем на 20% от начального
- [ ] Средняя CPU-нагрузка при активном browsing (15+ вкладок Google) не превышает 5% за 5-минутное окно (измеряется через `(Get-Process geo-fix).CPU` в PowerShell или Process Explorer)

**Оптимизация (внутренние механизмы):**
- [ ] Completed flows удаляются из памяти прокси после обработки — количество хранимых flows не растёт со временем
- [ ] WebSocket message history обрезается после обработки — хранится не более 1 последнего сообщения
- [ ] При превышении RAM-порога (300MB Private Working Set) прокси автоматически перезапускается; cooldown между перезапусками не менее 10 минут; не более 3 перезапусков в час

**Сохранение функциональности:**
- [ ] Все существующие тесты (unit, integration, E2E) проходят без изменений
- [ ] Accept-Language header rewriting работает на target domains (Google properties) и любых других доменах, проходящих через прокси
- [ ] JS injection работает на target domains — injected script присутствует в HTML-ответах Google
- [ ] CSP header содержит nonce injected скрипта — браузер не блокирует его выполнение
- [ ] При крэше основного процесса watchdog выполняет cleanup в течение 10 секунд
- [ ] Country switching через tray menu переключает preset без рестарта прокси

## Ограничения

- Все текущие функции должны быть полностью сохранены — это рефакторинг, не редизайн
- Уровень безопасности не должен снижаться (CA key lifecycle, DPAPI state encryption, session isolation)
- Watchdog должен оставаться надёжным — экономия 15MB не оправдывает снижение надёжности
- Приложение должно работать на Windows 10/11 с Python 3.12+
- PyInstaller build должен продолжать работать
- Совместимость с существующими desktop shortcuts и install.bat

## Риски

- **Риск 1:** Удаление "лишних" mitmproxy аддонов может сломать неочевидные сценарии (CONNECT tunneling для non-HTTP трафика, HTTP/2 negotiation). **Митигация:** Тщательно определить минимальный набор необходимых аддонов через изучение кода mitmproxy, сохранить все аддоны связанные с прокси-функциональностью (Core, ProxyServer, NextLayer, TlsConfig), протестировать на реальных сценариях.
- **Риск 2:** Немедленное удаление flow из View после обработки может конфликтовать с внутренней логикой mitmproxy. **Митигация:** Добавить flow cleanup как последний аддон в цепочке (после GeoFixAddon), тестировать с длительными сессиями.
- **Риск 3:** Обрезание WebSocket messages может нарушить WebSocket-зависимые сервисы (Google Docs realtime, Meet). **Митигация:** Сохранять последнее сообщение, не трогать активные соединения до завершения.
- **Риск 4:** Авто-рестарт прокси при превышении RAM-порога может оборвать активные запросы/WebSocket-соединения. **Митигация:** Cooldown 10 минут между рестартами, максимум 3 в час; если лимит превышен — только логирование без рестарта. Для пользователя рестарт выглядит как кратковременная (~5 сек) потеря связи — браузер автоматически переподключается к прокси и WebSocket-соединениям.
- **Риск 5:** Базовый footprint mitmproxy + Python runtime (~50-60MB) может не позволить уложиться в 100MB. **Митигация:** Целевой диапазон 100-150MB; если baseline после оптимизации >120MB — принять как нижнюю границу и сфокусироваться на стабильности (отсутствие роста).

## Технические решения

- Мы решили очищать обработанные flows из памяти прокси сразу после обработки, потому что прокси по умолчанию хранит все flows бессрочно — главный источник роста RAM.
- Мы решили обрезать историю WebSocket-сообщений, потому что каждое WS-соединение (Google Docs, Meet) накапливает все сообщения за сессию без ограничений.
- Мы решили минимизировать количество загружаемых компонентов прокси, потому что из ~33 дефолтных для geo-fix нужны только ~5 (прокси-ядро, TLS, маршрутизация).
- Мы решили не заменять watchdog subprocess на batch-скрипт, потому что надёжность crash recovery важнее 15MB экономии.
- Мы решили добавить защитный механизм: мониторинг RAM с перезапуском прокси при превышении 300MB (cooldown 10 минут, макс 3 раза в час), потому что пользователь готов к ~5 сек перерыву ради стабильности при долгой работе.
- Мы решили не использовать streaming для больших ответов, потому что JS injection требует доступа к полному телу HTML-ответа, а non-target домены и так не обрабатываются.

## Тестирование

**Unit-тесты:** делаются всегда, не обсуждаются.

**Интеграционные тесты:** делаем — нужно убедиться что оптимизированный прокси корректно обрабатывает реальные HTTP/HTTPS/WebSocket потоки, flow cleanup не ломает pipeline, и все существующие интеграционные тесты проходят.

**E2E тесты:** делаем — критично проверить что JS injection, CSP nonce, и header rewriting работают через оптимизированный прокси в реальном браузере. Существующие E2E тесты должны проходить без изменений.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Запуск существующих unit-тестов | bash: `pytest test/ -x` | Все тесты проходят |
| 2. Запуск интеграционных тестов | bash: `pytest test/ -m integration` | Все тесты проходят |
| 3. Проверка flow cleanup через unit-тест | bash: `pytest test/ -k flow_clean` | Тест подтверждает что flows удаляются после response/error |
| 4. Проверка минимизации аддонов | bash: `grep -n "addons.add\|default_addons" src/main.py` | Загружается минимальный набор аддонов |

### Пользователь проверяет
- Запустить geo-fix US, открыть 15+ вкладок Google, работать 2+ часа — RAM в Task Manager должна быть стабильной ~100-150MB
- Открыть Google Docs и редактировать документ 30+ минут — WebSocket-соединение не должно раздувать RAM
- Проверить что все функции работают: country switching, tray menu, --stop, запуск после крэша
