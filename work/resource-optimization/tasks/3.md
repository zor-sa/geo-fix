# Task 3: Add RAM monitoring with proxy auto-restart

## Required Skills

Before executing this task, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Extend the existing `_monitor_loop` in `src/main.py` to check process memory (Private Working Set) every 60 seconds and restart the mitmproxy thread if it exceeds 300MB. The restart is guarded by: a 10-minute cooldown between restarts, a rate limit of max 3 restarts per hour (then log-only), and an idle guard that defers restart if any flow was processed within the last 10 seconds.

Memory is read via Windows `ctypes` `GetProcessMemoryInfo` (`PROCESS_MEMORY_COUNTERS_EX.PrivateUsage`). On Linux/non-Windows, falls back to parsing `VmRSS` from `/proc/self/status`.

The restart sequence must regenerate and reinstall the CA certificate (original key files were deleted from disk as a security measure), then reuse the same `GeoFixAddon` instance to preserve current preset state and JS payload cache.

This task depends on Task 1 (minimal Master) and Task 2 (FlowCleanup addon), which introduce the `_start_mitmproxy` refactor and the `FlowCleanup` class that must be re-added after restart.

## What to do

1. Add `_last_flow_time` tracking to `GeoFixAddon` in `src/proxy_addon.py` — update it via `time.monotonic()` in the `request()` hook.

2. Add memory-reading helpers in `src/main.py`:
   - `_get_private_bytes_windows()` using `ctypes` `GetProcessMemoryInfo` → `PROCESS_MEMORY_COUNTERS_EX.PrivateUsage`
   - `_get_private_bytes_linux()` parsing `VmRSS` from `/proc/self/status`
   - `_get_process_memory_mb()` dispatcher: Windows → windows helper, else → linux helper, returns float MB

3. Add restart state variables at module level (or scoped to `main()`) to track cooldown and rate limiting:
   - `_last_restart_time: float = 0.0`
   - `_restart_timestamps: list[float] = []` (timestamps of restarts in current hour)

4. Add `_restart_mitmproxy()` function implementing the full restart sequence:
   - `master.shutdown()` (call on old master — requires refactoring `_start_mitmproxy` to expose/return `master`)
   - Create new `Master` with same `opts` (same `confdir=session_tmpdir`, same `port`)
   - `uninstall_ca_cert(thumbprint=state.ca_thumbprint)` — actual signature: `uninstall_ca_cert(thumbprint: Optional[str] = None)`; pass the stored thumbprint from the current state
   - `install_ca_cert(session_tmpdir)` → get new thumbprint
   - `delete_ca_key_files(session_tmpdir)` + `delete_ca_public_cert(session_tmpdir)`
   - Update `state.ca_thumbprint`, call `save_state(state)`
   - Re-add same `GeoFixAddon` instance + new `FlowCleanup()` to the new master
   - Start new master in new daemon thread

5. Extend `_monitor_loop` with RAM check logic after the existing VPN/watchdog checks:
   - Read memory via `_get_process_memory_mb()`
   - If below threshold (300MB): continue
   - If above threshold: check idle guard (`addon._last_flow_time` — skip if flow within last 10 sec)
   - Check cooldown (10 min since `_last_restart_time`)
   - Check rate limit (fewer than 3 entries in `_restart_timestamps` within last 3600 sec)
   - If all guards pass: call `_restart_mitmproxy()`, update `_last_restart_time` and `_restart_timestamps`
   - If rate limit exceeded: log warning only, no restart

## Details

**Files:**
- `src/main.py` — primary file to modify: memory helpers, restart function, monitor loop extension. Note: Task 1 already changed `_start_mitmproxy()` to return `(thread, master)` — use the `master` reference for `master.shutdown()` on restart
- `src/proxy_addon.py` — add `_last_flow_time: float = 0.0` attribute and `self._last_flow_time = time.monotonic()` in `request()` hook (thread-safe: only written from mitmproxy event loop, read from monitor thread — float assignment is atomic in CPython, but use a lock to be safe)
- `test/test_ram_monitor.py` — new test file

**Master and addon reference access:**
`_restart_mitmproxy()` cannot be a standalone function that captures `master` and `addon` from closure — `_monitor_loop`'s closure locals are not accessible to it. Instead, store both as accessible references: either as instance variables (if `main()` logic is wrapped in a class) or as module-level references updated each time `_start_mitmproxy` is called. The `GeoFixAddon` instance created before the first `_start_mitmproxy` call and the `master` object returned by it must both be reachable from `_restart_mitmproxy` at restart time.

**Dependencies:**
- Task 1 must be complete: `_start_mitmproxy` already uses base `Master` (not `DumpMaster`), which is required for the restart sequence
- Task 2 must be complete: `FlowCleanup` class exists in `src/proxy_addon.py` and must be re-instantiated and re-added after each restart

**Edge cases:**
- `GetProcessMemoryInfo` call failure on Windows (permissions, API unavailable): catch `OSError`, log warning, skip threshold check for this cycle
- `/proc/self/status` missing on Linux (non-standard kernel): catch `FileNotFoundError`, return 0.0 (skip check)
- `master.shutdown()` blocks or raises during restart: catch exception, log error, mark restart failed, do not update timestamps (avoid false restart count)
- `install_ca_cert()` returns `None` (certutil failure): log error, do not proceed with starting new master — old proxy is already down; handle as restart failure, retry on next cycle
- Rate limit window: `_restart_timestamps` should only keep entries within the last 3600 seconds — prune on each check

**Implementation hints:**
- Use `ctypes` to define a `PROCESS_MEMORY_COUNTERS_EX` structure with `SIZE_T` fields (see MSDN for the full field list); the target field is `PrivateUsage`. Call `GetProcessMemoryInfo` with the current process handle obtained from `kernel32.GetCurrentProcess()`.
- Decision 3: RAM monitor in existing thread (not new thread) — just add after VPN/watchdog checks inside `_monitor_loop`
- Decision 4: thread restart (not process restart) — tray, watchdog, and system proxy settings are preserved
- Decision 6: no psutil — ctypes only
- Task 1 already returns `(thread, master)` from `_start_mitmproxy()` — use the `master` reference in the restart function to call `master.shutdown()`
- Restart sequence timing: shutdown (~2 sec) + new master start (~3 sec) + CA install (~2 sec) = ~7 sec total connectivity gap
