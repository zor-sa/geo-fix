# Task 5: Integration testing of optimized proxy

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Tasks 1–4 replace `DumpMaster` with minimal `Master`, add `FlowCleanup`, add RAM monitoring with proxy auto-restart, and optimize hot-path CPU paths. This task verifies all these changes work correctly end-to-end: that the optimized proxy correctly routes HTTP and HTTPS/CONNECT traffic, that JS injection still works through the minimal addon chain, that `FlowCleanup` does not cause memory growth, and that the proxy restart sequence preserves TLS functionality.

The integration tests live in a new file `test/test_integration_proxy.py`. They complement `test/test_integration_windows.py` (which is Windows-only and tests registry/CA cert operations) — the new tests focus on proxy traffic correctness across platforms using mitmproxy's internal API rather than subprocess curl.

## What to do

1. Create `test/test_integration_proxy.py` with a `pytest` test module
2. Implement a shared fixture that starts the optimized proxy (minimal `Master` + `GeoFixAddon` + `FlowCleanup`) on a free port, yields control to the test, then shuts it down
3. Write test for **HTTP traffic** — send a plain HTTP request through the proxy, verify a response arrives
4. Write test for **HTTPS/CONNECT tunneling** — send a CONNECT request, verify the tunnel is established
5. Write test for **JS injection** — send an HTTP request to a target domain (mock server or httpbin), verify the injected `<script nonce=` tag appears in the response body
6. Write test for **flow cleanup / no memory growth** — process 100 flows through the proxy, then assert zero flow objects are retained
7. Write test for **proxy restart preserves TLS** — restart the mitmproxy thread (simulate the RAM-triggered restart path from `_start_mitmproxy`), then send a request and verify the proxy is still accepting connections after restart
8. Ensure all existing tests in `test/test_integration_windows.py` continue to pass unchanged (no modifications to that file)

## TDD Anchor

Write tests before implementation. Since this task **is** the tests (no production code), the anchor is: write each test, run it, confirm it fails or is skipped for the right reason, then make it pass (adjusting only fixture/helper code, not production code).

- `test/test_integration_proxy.py::TestOptimizedProxyHTTP::test_http_request_proxied` — start minimal-Master proxy, send HTTP GET, verify 200 response arrives through proxy
- `test/test_integration_proxy.py::TestOptimizedProxyHTTPS::test_connect_tunnel_established` — send CONNECT request, verify TCP tunnel established (no SSL error on socket level)
- `test/test_integration_proxy.py::TestJSInjection::test_js_injected_for_target_domain` — proxy returns HTML with `<script nonce=` for a google.com target domain request
- `test/test_integration_proxy.py::TestFlowCleanup::test_no_flows_retained_after_100_requests` — after 100 proxied requests, flow count is 0
- `test/test_integration_proxy.py::TestProxyRestart::test_tls_works_after_restart` — proxy restarts (thread stop + new thread start), proxy accepts connections again within 15 seconds

## Details

**Files:**
- `test/test_integration_proxy.py` (new) — all integration tests for the optimized proxy

**Dependencies:**
- Task 3 must be done: `src/main.py` has the new minimal-Master `_start_mitmproxy()` and proxy restart logic
- Task 4 must be done: `src/proxy_addon.py` has `FlowCleanup` class

**Starting the optimized proxy in tests:**

Use the same pattern as `test_integration_windows.py::TestProxyStartsAndInjects`:
- Start a new `asyncio` event loop in a daemon thread
- Use `mitmproxy.master.Master` (not `DumpMaster`) with minimal addons — exactly the same construction as `_start_mitmproxy()` post-Task-1
- Add `GeoFixAddon` + `FlowCleanup` to the master
- Wait for port to be open (poll with `socket.create_connection`)
- Use a unique port (e.g., `18092`+) to avoid collisions with `test_integration_windows.py` (port 18090/18091)
- Tear down by calling `master.shutdown()` after the test

**Sending HTTP requests in tests:**

Use Python's built-in `urllib.request` with `ProxyHandler` — no external dependencies:
```python
proxy_handler = urllib.request.ProxyHandler({"http": "http://127.0.0.1:{port}"})
opener = urllib.request.build_opener(proxy_handler)
response = opener.open("http://httpbin.org/headers", timeout=10)
```
If `httpbin.org` is unreachable (CI may have no network), fall back to `pytest.skip()` — same pattern used in `test_integration_windows.py`.

**Testing JS injection without network:**

Use a local `http.server.HTTPServer` in a thread that serves a fake `www.google.com`-hosted HTML response. The proxy intercepts it, `GeoFixAddon.response()` runs, JS is injected. Assert `<script nonce=` in the response body. This avoids network dependency for the injection test.

Alternatively, mock the flow directly using the `FakeFlow` helper pattern from `test_proxy_addon.py` — call `addon.response(flow)` directly — but this bypasses the proxy pipeline. Prefer a real proxy pipeline for this integration test.

**Testing flow cleanup:**

After 100 flows, inspect `master.state.flows` (or equivalent mitmproxy internal). With minimal `Master` and no `View` addon loaded, the flow store should be empty (flows are never retained). `FlowCleanup` additionally clears content. Assert `len(master.state.flows) == 0` or the equivalent API.

**Testing proxy restart:**

1. Start proxy, verify it accepts connections
2. Call `master.shutdown()` on the current master
3. Start a new `Master` (same port) via `_start_mitmproxy()` or equivalent
4. Poll for port to be open (up to 15 seconds)
5. Send a request and verify it succeeds
No actual TLS cert install/uninstall needed — just verify the proxy thread restarts and accepts connections.

**Edge cases:**
- Port collisions: use `_select_port()` or hardcode ports above 18090 that won't conflict
- Async event loop isolation: each test that starts a proxy must create its own `asyncio` event loop (`asyncio.new_event_loop()` + `asyncio.set_event_loop(loop)`) in the proxy thread, same pattern as existing integration tests
- `master.shutdown()` is async — call it via `loop.call_soon_threadsafe(loop.stop)` or `asyncio.run_coroutine_threadsafe(master.shutdown(), loop)` depending on mitmproxy version
- Graceful teardown: always shut down the proxy in a `try/finally` block inside a fixture to prevent port leaks between tests
- `pytest.skip()` on network-dependent tests when external host is unreachable (check `returncode` or `except` on connection error)
- Windows vs Linux: no registry/certutil calls in these tests — all tests should run on both platforms

**Implementation hints:**
- Copy the startup pattern verbatim from `test_integration_windows.py::TestProxyStartsAndInjects::run_proxy()` — it handles `asyncio` event loop setup correctly for mitmproxy
- Look at `src/main.py::_start_mitmproxy()` for the exact addon list to replicate (after Task 1 changes)
- `FlowCleanup` is imported from `src.proxy_addon` (after Task 2)
- For the local mock server (JS injection test): use `threading.Thread(target=httpd.serve_forever, daemon=True)` + `httpd.shutdown()` in cleanup
