# Task 8: Test Audit

## Required Skills

Before starting, load:
- `/skill:test-master` — [SKILL.md](~/.claude/skills/test-master/SKILL.md)

## Description

Full-feature test quality audit for the resource-optimization feature. This task reads all test files — both existing and newly created — to assess whether the test suite adequately covers the three main changes introduced by this feature: the minimal Master setup (Task 1), the FlowCleanup addon (Task 2), and the RAM monitor with auto-restart (Task 3). It also verifies that existing tests for original GeoFixAddon functionality were not regressed or accidentally weakened during the feature work.

The audit produces a written report documenting coverage gaps, test quality issues, and any recommendations. Unlike implementation tasks, this task does not write production code — it writes a report as the output artifact.

Note: `test/test_integration_proxy.py` is a new file created by Task 5 and may not exist yet when this task begins. If Task 5 is not yet complete, read whatever is available and note the gap.

## What to do

1. **Load the skill** — `/skill:test-master` before starting.

2. **Read all test files** relevant to this feature:
   - `test/test_proxy_addon.py` — existing unit tests for GeoFixAddon and helpers
   - `test/test_integration_windows.py` — existing integration tests (Windows proxy, CA cert, state file, single-instance guard)
   - `test/test_integration_proxy.py` — new integration tests from Task 5 (HTTP/HTTPS/WebSocket flows, flow cleanup, proxy restart)
   - Any new unit test files created by Tasks 1–4 (e.g., `test/test_master_setup.py`, `test/test_ram_monitor.py`, `test/test_flow_cleanup.py`)

3. **Audit coverage** against the three new components:
   - **FlowCleanup addon** — are `response`, `error`, `websocket_message`, `websocket_end` hooks all tested? Is the "added after GeoFixAddon" ordering verified?
   - **RAM monitor** — are threshold detection, cooldown, rate limiting (3/hour cap), idle guard (both defer and proceed cases), and Linux fallback path all covered?
   - **Minimal Master setup** — is there a test verifying `Master` is used instead of `DumpMaster`? Are the 6 required addons (Core, Proxyserver, NextLayer, TlsConfig, KeepServing, ErrorCheck) verified to be present?

4. **Audit regression coverage** — verify the existing `test/test_proxy_addon.py` tests still cover all original GeoFixAddon behavior: Accept-Language rewriting, JS injection, CSP nonce modification, preset switching, non-target domain skipping, non-HTML skipping, large response skipping, non-200 skipping.

5. **Check test quality** for each area:
   - Are mocks/fakes realistic or do they underspecify the behavior being tested?
   - Are edge cases covered (None guards, empty bodies, concurrent access)?
   - Are integration tests actually starting the proxy or only mocking it?
   - Is there a test that verifies no requests leave the machine during proxy restart (Decision 4 safety requirement)?

6. **Write the audit report** — as `## Audit Report` section at the bottom of this task file (or as a separate file `logs/working/task-8/test-audit.md` if large). Include:
   - Coverage verdict per component (Covered / Partially covered / Not covered)
   - Specific missing tests with test IDs and what they should verify
   - Test quality issues found (with file + line references)
   - Overall pass/fail verdict: does the suite meet the Testing Strategy in the tech-spec?
