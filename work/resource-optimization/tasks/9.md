# Task 9: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Приёмочное тестирование фичи resource-optimization перед деплоем. Запускаем полный тест-сьют (unit + integration + E2E), проверяем все критерии приёмки из user-spec и tech-spec, подтверждаем отсутствие регрессий.

Это финальная волна после аудитов (tasks 6, 7, 8) — всё уже реализовано и проверено на качество. Задача QA: убедиться что реализация соответствует требованиям пользователя и техническим критериям.

## What to do

1. Запустить полный тест-сьют: `pytest test/ -x -v`
2. Пройти по каждому пункту acceptance criteria из user-spec и tech-spec — отметить выполненные
3. Зафиксировать результаты в QA-отчёте
4. Если какой-то критерий не выполнен — зафиксировать как блокер

## Acceptance Criteria

**Из user-spec (ресурсы):**
- [ ] Базовое потребление RAM при старте не превышает 150MB (основной процесс + watchdog)
- [ ] RAM стабильна после 8 часов работы — Private Working Set не растёт более чем на 20% от начального
- [ ] Средняя CPU-нагрузка при активном browsing (15+ вкладок Google) не превышает 5% за 5-минутное окно

**Из user-spec (оптимизация):**
- [ ] Completed flows удаляются из памяти прокси после обработки — количество хранимых flows не растёт со временем
- [ ] WebSocket message history обрезается после обработки — хранится не более 1 последнего сообщения
- [ ] При превышении RAM-порога (300MB) прокси автоматически перезапускается только если нет активного трафика (>10 сек без запросов); cooldown между перезапусками не менее 10 минут; не более 3 перезапусков в час

**Из user-spec (сохранение функциональности):**
- [ ] Все существующие тесты (unit, integration, E2E) проходят без изменений
- [ ] Accept-Language header rewriting работает на target domains и любых других доменах
- [ ] JS injection работает на target domains — injected script присутствует в HTML-ответах Google
- [ ] CSP header содержит nonce injected скрипта — браузер не блокирует его выполнение
- [ ] При крэше основного процесса watchdog выполняет cleanup в течение 10 секунд
- [ ] Country switching через tray menu переключает preset без рестарта прокси

**Из tech-spec (технические критерии):**
- [ ] mitmproxy стартует с base Master class, не DumpMaster — unused addons не загружены
- [ ] Загружены только essential addons: Core, Proxyserver, NextLayer, TlsConfig, KeepServing, ErrorCheck, GeoFixAddon, FlowCleanup
- [ ] FlowCleanup добавлен после GeoFixAddon — подтверждено unit-тестом проверки порядка аддонов
- [ ] FlowCleanup очищает flow content после обработки — flow.request.content и flow.response.content установлены в None/empty
- [ ] WebSocket message history обрезается до <=1 сообщения на соединение
- [ ] RAM monitoring срабатывает при 300MB Private Working Set с cooldown 10 мин, макс 3/час
- [ ] Proxy restart происходит только после 10+ секунд без flow activity (idle guard)
- [ ] Proxy restart переиспользует тот же экземпляр GeoFixAddon (сохраняет текущий preset и JS payload cache)
- [ ] CA сертификат пересоздаётся, переустанавливается и ключ удаляется при proxy restart — HTTPS работает после restart
- [ ] Во время proxy restart HTTP-запросы не покидают машину (браузер получает connection refused, не direct fallback)
- [ ] Нет регрессий в существующем тест-сьюте (unit + integration + E2E)
- [ ] Dumper addon не загружен (нет stdout output per flow)

## Verification Steps

### Automated

- `pytest test/ -x -v` → all tests pass (unit + integration + E2E)
- `pytest test/ -k flow_clean` → tests confirm flows cleared after response/error
- `grep -n "addons.add\|default_addons" src/main.py` → only minimal addon set loaded, no DumpMaster

## Details

**Dependencies:** Tasks 6 (code audit), 7 (security audit), 8 (test audit) must be complete before running QA

**Edge cases:**
- Проверь что тесты для RAM monitor покрывают все ветки: threshold detection, cooldown, rate limit (4-й рестарт подавляется), idle guard (defer when active), idle guard (proceed when idle)
- Проверь что интеграционный тест подтверждает: при системном прокси, указывающем на non-listening port, браузер NOT отправляет direct requests

**Implementation hints:**
- Критерии ресурсного потребления (RAM <150MB, CPU <5%) — не могут быть проверены автоматически агентом, отмечаются как "requires user verification" в отчёте
- Для критерия "no direct fallback during proxy restart" — найди соответствующий интеграционный тест и убедись что он существует и проходит
- Результаты сохрани в `logs/working/task-9/qa-report.md`
