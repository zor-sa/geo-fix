# Task 2: Add FlowCleanup addon

## Required Skills

Before executing this task, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Add a `FlowCleanup` addon to `src/proxy_addon.py` that clears flow content (request/response bodies) after processing to reduce GC pressure, and trims WebSocket message history to prevent unbounded memory growth during long sessions (Google Docs, Meet).

The addon must be registered last in the addon chain (after `GeoFixAddon`) in `src/main.py` so that GeoFixAddon finishes processing flows before cleanup runs. Decision 2 in the tech-spec explains why this is a separate addon rather than merged into GeoFixAddon: separation of concerns makes it independently testable.

Currently `main.py` uses `DumpMaster` (Task 1 changes that to minimal `Master`). This task only concerns `proxy_addon.py` — adding the `FlowCleanup` class. The instruction to add it after `GeoFixAddon` in the addon chain must also be reflected in `main.py` (currently `master.addons.add(addon)` line).

## What to do

1. Add a `FlowCleanup` class to `src/proxy_addon.py` with these hooks:
   - `response(flow)` — clears `flow.request.content` and `flow.response.content` (set to `b""` or `None`) after GeoFixAddon has finished modifying the response
   - `error(flow)` — clears `flow.request.content` to free memory for errored flows
   - `websocket_message(flow)` — trims `flow.websocket.messages` to keep only the last 1 message
   - `websocket_end(flow)` — removes the flow reference to release memory

2. Update `src/main.py` to add `FlowCleanup()` after `GeoFixAddon` in the addon chain (add `from src.proxy_addon import FlowCleanup` import and `master.addons.add(FlowCleanup())` call after the existing `master.addons.add(addon)`)

## Details

**Files:**
- `src/proxy_addon.py` — add `FlowCleanup` class after `GeoFixAddon`
- `src/main.py` — add `FlowCleanup` import and registration after `GeoFixAddon` in `_start_mitmproxy()`

**Dependencies:** None (Wave 1, independent of Task 1)

**Edge cases:**
- `flow.response` may be `None` in `response()` hook — guard before accessing `.content`
- `flow.websocket` may be `None` or messages list empty — guard before trimming
- `websocket_end()`: removing the flow from mitmproxy's internal state uses `flow` object deletion or setting content to None; check mitmproxy API for correct approach (`del flow` vs `flow.kill()` vs content clearing)

**Implementation hints:**
- Follow the same hook method signatures as `GeoFixAddon` — mitmproxy discovers hooks by method name, no registration required
- Setting `flow.request.content = b""` and `flow.response.content = b""` is sufficient to release body bytes; the flow object itself is managed by mitmproxy
- For `websocket_message()`: `flow.websocket.messages[-1:]` gives the last message; assign back as slice `flow.websocket.messages[:] = flow.websocket.messages[-1:]`
- `FlowCleanup` has no `__init__` state — it is stateless, so no `threading.Lock` needed
- The addon chain order in `_start_mitmproxy()` must be: `master.addons.add(addon)` (GeoFixAddon) then `master.addons.add(FlowCleanup())` — this matches the architecture diagram in the tech-spec
