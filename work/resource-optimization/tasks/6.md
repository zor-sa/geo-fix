## Description

Full-feature code quality audit for the resource-optimization feature. This task reviews all three source files modified in this feature (src/main.py, src/proxy_addon.py, src/presets.py) for cross-component correctness, integration risks, and code quality issues introduced by the feature.

The feature makes non-trivial architectural changes: replacing DumpMaster with a minimal Master, adding a FlowCleanup addon after GeoFixAddon, and extending the monitor thread with RAM watchdog + proxy auto-restart (including CA cert regeneration). These changes span multiple components with shared state, thread interactions, and a complex restart sequence — making a cross-component audit essential before pre-deploy QA.

## What to do

1. Read all three source files in full: src/main.py, src/proxy_addon.py, src/presets.py
2. Review for correctness of the minimal Master setup — verify essential addons are loaded in the right order, no required addon is missing for HTTPS/CONNECT/WebSocket flows
3. Review FlowCleanup addon integration — addon ordering relative to GeoFixAddon, correctness of content clearing, WebSocket trim logic, edge cases
4. Review the RAM monitor and proxy restart sequence — thread safety, restart idempotency, CA cert lifecycle correctness (delete → regenerate → reinstall → delete), state update atomicity, rate limiting logic
5. Review idle guard implementation — `_last_flow_time` tracking in GeoFixAddon, thread safety of the check in monitor, race condition risks
6. Review cross-component issues: references between main.py and proxy_addon.py, shared GeoFixAddon instance reuse across restarts, state consistency
7. Review presets.py for any issues introduced by Task 4 CPU optimizations (tuple-based endswith, case-insensitive search changes)
8. Write the audit report as the output of this task (see Acceptance Criteria)

## Details

**Files to read:**
- `src/main.py` — Focus areas: `_start_mitmproxy()` (minimal Master setup, addon loading order), `_monitor_loop()` (RAM check logic, restart trigger conditions, idle guard check), restart sequence steps, `_do_cleanup()` interaction with restart
- `src/proxy_addon.py` — Focus areas: `FlowCleanup` class (response/error/websocket_message/websocket_end hooks), `GeoFixAddon._last_flow_time` tracking, thread safety of `switch_preset()` after restart
- `src/presets.py` — Focus areas: `is_target_domain()` after tuple-based endswith optimization, `_build_js_payload()` language list construction edge cases

**Key cross-component risks to examine:**

1. **Addon ordering**: The tech-spec requires FlowCleanup to be added AFTER GeoFixAddon. If this order is wrong, FlowCleanup.response() runs before GeoFixAddon.response() — JS injection content would be cleared before the browser receives it.

2. **GeoFixAddon reuse across restart**: The same `addon` instance is passed to the new Master after restart. Verify that the GeoFixAddon holds no references to the old Master or old flow objects that would prevent GC of the old master.

3. **CA cert lifecycle on restart**: The restart sequence must: (1) shut down old master, (2) create new Master (which generates new CA in confdir), (3) delete old CA from cert store, (4) install new CA, (5) delete new CA key files from disk. Any deviation from this order could leave the system with no valid CA (browser TLS fails) or with CA keys on disk (security regression).

4. **Idle guard thread safety**: `GeoFixAddon._last_flow_time` is written by the mitmproxy async event loop (in `request()` hook) and read by the monitor thread. Without a lock or atomic operation, there may be a race. Assess whether this is a real risk given Python's GIL and `time.monotonic()` semantics.

5. **Rate limiting state**: The restart counter and last-restart timestamp must survive across restart cycles (they should be in the monitor thread's local scope, not in the master). Verify they are not reset when the master restarts.

6. **Monitor thread and stop_event**: The monitor thread uses `stop_event.wait(timeout=60)`. Verify that when a restart is triggered mid-wait, the next iteration correctly picks up the new master reference.

7. **`_find_inject_position` optimization (Task 4)**: If Task 4 changed this function to avoid full `.lower()` copy, verify the case-insensitive search still correctly finds `<head`, `<html`, `<!doctype` in mixed-case HTML.

8. **`is_target_domain` tuple optimization (Task 4)**: If changed to `host.endswith(tuple(...))`, verify the exact-match branch (`host == domain.lstrip(".")`) is preserved for bare domain matching.

**Severity guide:**
- **Critical**: Would cause data loss, security regression, or complete proxy failure
- **Major**: Would cause incorrect behavior in common scenarios (TLS broken after restart, JS not injected, memory not freed)
- **Minor**: Incorrect behavior in edge cases, or missing error handling that could surface as confusing failures
- **Nitpick**: Style, naming, missing log messages, redundant code
