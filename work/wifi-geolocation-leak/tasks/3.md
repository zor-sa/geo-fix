## Description

Wire Location Services disable/restore into the main.py startup sequence and cleanup flow. This task connects the registry functions from Task 1 (`disable_location_services` / `restore_location_services`) to the application lifecycle:

1. Add `original_location_services: Optional[str] = None` field to `ProxyState` so the pre-disable registry value survives a crash and enables correct restore.
2. Call `disable_location_services()` early in the startup sequence in `main()`, store the returned original value in `state.original_location_services`, and save state.
3. Add a new cleanup label `"Location Services restore"` to `_VALID_CLEANUP_LABELS` and `_execute_cleanup_by_label()` in system_config.py so it participates in the existing retry/pending resilience pattern.
4. Call `restore_location_services()` inside `cleanup()`, using the stored original value from state.

This task depends on Task 1 (the registry functions must exist before they can be wired). No changes to proxy_addon.py or inject.js — those are Task 2.

## What to do

1. Add `original_location_services: Optional[str] = None` to `ProxyState` in system_config.py. The field must have a default so existing state files (without this field) deserialize correctly — `from_json` uses strict schema validation, which means old files that don't include this field will still load because `**parsed` will simply omit it and the default applies.

2. Add `CLEANUP_LABEL_LOCATION_SERVICES = "Location Services restore"` constant alongside the other label constants in system_config.py.

3. Add that label to `_VALID_CLEANUP_LABELS` frozenset.

4. Add a branch to `_execute_cleanup_by_label()` for the new label: call `restore_location_services(original=None)` (stateless — no stored value available in pending cleanup context).

5. Add the Location Services restore step to `cleanup()` after firewall removal and before `delete_state()`. Use `_try_step(CLEANUP_LABEL_LOCATION_SERVICES, restore_location_services, state.original_location_services)`.

6. Import `disable_location_services` and `restore_location_services` in main.py (add to the existing `from src.system_config import (...)` block).

7. In `main()`, call `disable_location_services()` after `create_firewall_rules()` and before the tray is started. Store the return value in `state.original_location_services` and call `save_state(state)`.

8. Write the TDD anchor tests (see below) before implementing, verify they fail, then implement, verify they pass.

**Dependencies:** Task 1 must be complete — `disable_location_services()` and `restore_location_services()` must exist in system_config.py before this task can be implemented.

**Edge cases:**
- Non-Windows: `disable_location_services()` is a no-op per Task 1 implementation, returns `None`. ProxyState stores `None`, cleanup calls `restore_location_services(None)` which is also a no-op. No special handling needed in this task.
- Crash before `state.original_location_services` is saved: cleanup via watchdog reads state from disk — if original value was not yet written, field defaults to `None`, and `restore_location_services(None)` deletes the key (safe behavior per Decision 4).
- Pending cleanup from previous session: `_execute_cleanup_by_label("Location Services restore")` calls `restore_location_services(original=None)` (stateless — original value not available), which deletes the key. This is the safe default.
- `from_json` strict schema validation: adding the new field with a default means old state files (without the key) still pass validation — `**parsed` simply doesn't supply it, the dataclass default kicks in. New state files (with the key) also pass. No migration needed.

**Implementation hints:**
- The `from_json` strict schema validation checks for *unknown* fields, not *missing* fields. Adding a new field with a default to the dataclass means old files (missing the field) load fine; `**parsed` omits the key and the default `None` applies. This is the existing backward-compat mechanism — `firefox_prefs_backup`, `session_id`, etc. all use the same pattern.
- In `cleanup()`, the Location Services restore should happen after firewall removal and before `delete_state()`. Rationale: registry restore doesn't depend on any state file content other than the stored original value, which is read directly from the `state` object already in memory.
- `restore_location_services` signature from Task 1 accepts `original: Optional[str]`. Pass `state.original_location_services` directly.
- Keep the import block in main.py alphabetically sorted to match existing style.
