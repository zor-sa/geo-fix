# Task 5: Security Audit

## Required Skills

Before starting, load: `/skill:security-auditor` — [SKILL.md](~/.claude/skills/security-auditor/SKILL.md)

## Description

Full-feature security audit across all source files created or modified in this feature: `system_config.py`, `proxy_addon.py`, `inject.js`, and `main.py`. This feature adds three new attack surfaces: HKCU registry writes, proxy-level request interception, and universal JS injection. Analyze holistically against OWASP Top 10 and look at cross-component auth/data flows.

Key security concerns for this feature:
- Registry write/restore: can a tampered state file force arbitrary registry values? (input validation on restore)
- Proxy intercept: can an attacker craft a request that causes unintended intercept or response fabrication?
- JS injection broadened to all domains: does this create new CSP bypass or content injection vectors?
- `navigator.permissions.query` override: side-effects on non-geolocation permissions?
- Cleanup label allowlist: new label added safely without expanding the allowlist beyond what is needed?

## Details

**Security focus per file:**

- `src/system_config.py` — Registry write attack surface:
  - `restore_location_services()`: verifies original value is in `{'Allow', 'Deny', None}` — a tampered `.geo-fix-state.bin` must not be able to force arbitrary registry values. Confirm validation is enforced before write.
  - `disable_location_services()`: graceful fallback on missing registry path must not swallow genuine errors silently
  - Cleanup label: new label added to allowlist only, not expanding the allowlist beyond what is necessary (CWE-20 per architecture.md)
  - State file: `original_location_services` field validated on load — check schema validation covers this new field

- `src/proxy_addon.py` — Proxy intercept attack surface:
  - Geolocation API intercept: match is `host == 'www.googleapis.com'` AND `path == '/geolocation/v1/geolocate'` AND `method == 'POST'` — verify exact equality checks, not substring or prefix match (prevents bypass via crafted URLs)
  - Fabricated response: synthetic JSON must not include user-controlled content (coordinates come from preset, accuracy is random — no injection vector)
  - Broadened JS injection: injecting `<script>` into all HTML responses — ensure nonce is cryptographically random (existing `secrets.token_urlsafe` pattern), CSP modification does not inadvertently weaken existing policies
  - `script-src 'none'` skip guard: verify it truly skips injection and does not fall through

- `src/inject.js` — JS injection content security:
  - `navigator.permissions.query` override: must only intercept `name === 'geolocation'` — other permissions (camera, microphone, notifications) must pass through to original. Verify no over-broad interception.
  - Geolocation-only payload for non-target domains: verify it does not include timezone/language overrides that could interfere with non-Google sites

- `src/main.py` — Startup security:
  - `disable_location_services()` error handling: if registry call fails, must not abort startup with unhandled exception leaking system info
  - Original value stored in ProxyState: DPAPI-encrypted state file — no plaintext exposure risk

**Cross-component flow to audit:**
1. Attacker tampers `.geo-fix-state.bin` → `restore_location_services()` reads `original_location_services` → value validation → registry write. Is arbitrary value blocked?
2. Attacker sends crafted POST to `www.googleapis.com/geolocation/v1/geolocate` through proxy → intercept triggers → fake response. Any injection vector in the fabricated JSON?
3. Site with malicious CSP headers → proxy processes CSP → injection decision. Can a site force injection through `script-src 'none'` skip guard?

**Report format:** JSON with fields `task` (5), `findings` (array of `{severity, owasp_category, file, issue, fix}`), `summary` (string)
