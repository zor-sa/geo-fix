# Task 7: Fix firewall rules with auto-detected browser paths and unconditional lifecycle

## Required Skills
Before starting, load: /skill:code-writing

## Description
The current `create_firewall_rules()` uses a broken hardcoded pattern (`%ProgramFiles%\..\..\{browser}`) that resolves to invalid paths like `C:\chrome.exe`, so no traffic is ever blocked. Browser paths must be auto-detected from the Windows registry (`HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths\{exe}`) with a filesystem fallback to standard Program Files locations. Additionally, firewall rules are currently only created during the first-run wizard and only removed if `state.firewall_rules_created` is True — both conditions must be removed so rules are created on every geo-fix start and removed unconditionally on every cleanup.

## What to do

1. Add `_find_browser_path(exe_name: str) -> Optional[Path]` in `system_config.py`:
   - Query `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths\{exe_name}` default value via `winreg`.
   - If not found, try `HKLM\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\App Paths\{exe_name}`.
   - If still not found, check standard paths: `C:\Program Files\Google\Chrome\Application\chrome.exe`, `C:\Program Files (x86)\Google\Chrome\Application\chrome.exe`, `C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`, `C:\Program Files\Mozilla Firefox\firefox.exe`.
   - Return `None` if not found. On non-Windows, return `None`.
   - Mock boundary: wrap `winreg` import inside the function body for cross-platform testability.

2. Update `create_firewall_rules()`:
   - For each browser in `BROWSER_EXES`, call `_find_browser_path(browser)`.
   - If path is `None`: log `warning("Browser not found, skipping firewall rule: %s", browser)` and continue.
   - Use the resolved absolute path string in the `program=` argument of the `netsh` command (not the broken `%ProgramFiles%\..` pattern).
   - Remove the `firewall_rules_created` flag from `ProxyState` — this field must be deleted from the dataclass.

3. Update `cleanup()`:
   - Remove the `if state.firewall_rules_created:` guard — call `remove_firewall_rules()` unconditionally.
   - Remove any reference to the deleted `firewall_rules_created` field.

4. Update `main.py`:
   - Move `create_firewall_rules()` call into the main startup path (step 12 of the new startup sequence), not just inside the wizard.
   - Remove any wizard-only gating of firewall rule creation.

5. Remove `firewall_rules_created` from `ProxyState` dataclass and update `from_json` schema validation accordingly.

## Details
The current broken `program` path in the `netsh` command is `%ProgramFiles%\\..\\..\\{browser}` — this expands to something like `C:\Program Files\..\..\chrome.exe` = `C:\chrome.exe`. The fix must use an actual resolved path string. The `netsh advfirewall firewall add rule ... program=<path>` argument requires a non-expanded absolute path.

`firewall_rules_created: bool = False` must be removed from `ProxyState`. The `from_json` strict schema validation will reject any old state file containing this field — that is correct and expected behavior (old stale state files will be discarded on upgrade, which is acceptable).

On non-Windows test environments, `create_firewall_rules()` already logs a warning and returns `True` without calling netsh. The `_find_browser_path` function must also guard against importing `winreg` on non-Windows — use a try/import or `sys.platform` check inside the function, not at module level.
