# Task 5: Reorder startup — system changes only after mitmproxy bind, add port selection

## Required Skills
Before starting, load: /skill:code-writing

## Description
Currently in `main()`, `set_wininet_proxy()` and `set_firefox_proxy()` are called (lines 239–240) before `_start_mitmproxy()` is called (line 260). This means the system proxy points to a port that mitmproxy has not yet bound — if mitmproxy fails to start, all browser traffic is broken with no recovery. This task restructures the startup sequence so mitmproxy binds successfully before any system modifications occur. It also adds a `--port` CLI flag and automatic port selection via pre-bind socket, so that port conflicts are detected before anything is changed in the registry.

## What to do

1. **Add `--port` argument to `_parse_args()` in `src/main.py`** (lines 64–77):
   - `parser.add_argument("--port", type=int, default=None, help="Proxy port (default: 8080, auto-select if occupied)")`

2. **Add `_select_port(requested: Optional[int]) -> int` function to `src/main.py`**:
   - If `requested` is not None: attempt to bind a socket to `(PROXY_HOST, requested)`. If it binds, close the socket and return `requested`. If `OSError` (port busy), fall through to auto-select.
   - Auto-select: bind a socket to `(PROXY_HOST, 0)`, read `sock.getsockname()[1]`, close the socket, return that port.
   - If both fail (shouldn't happen with port 0 unless system is out of ports), raise `RuntimeError` with a clear message: "Cannot bind any proxy port — aborting. Registry not modified."
   - The pre-bind socket is closed before passing the port to mitmproxy (race window is negligible on localhost per tech-spec D-5).

3. **Reorder `main()` startup sequence** to match the tech-spec:

## Details

**Current startup order problem** in `main()` (lines 239–260):
```python
original_proxy = set_wininet_proxy()      # line 239 — registry modified HERE
firefox_backup = set_firefox_proxy()      # line 240 — Firefox modified HERE
state = ProxyState(...)                   # lines 243–253
save_state(state)                         # line 254
addon = GeoFixAddon(preset)               # line 257
proxy_thread = _start_mitmproxy(addon)    # line 260 — proxy starts LAST
```
The registry points to 127.0.0.1:8080 before mitmproxy is bound. If mitmproxy fails for any reason (port busy, import error, crash in addon initialization), the system is broken.

**`_start_mitmproxy()` hardcoding** (lines 154, 172): `listen_port=PROXY_PORT` and `check_proxy_running(PROXY_HOST, PROXY_PORT)` both use the constant. Must be updated to use the dynamic `port` parameter.

**`PROXY_PORT` constant** (line 28 of `system_config.py`): still useful as default value and for tests. Do not remove it — use as default parameter value: `def set_wininet_proxy(port: int = PROXY_PORT)`.

**`_select_port()` pre-bind approach**: binding to port 0 lets the OS assign a free ephemeral port. The socket is immediately closed after reading the port number. There is a tiny race (another process could grab that port in the ~milliseconds before mitmproxy binds), but this is negligible on localhost per tech-spec D-5.

**ProxyState `proxy_port` field**: needed so that on cleanup/restore, the correct `unset_wininet_proxy()` restores the registry properly. Without storing the port, cleanup always restores to `original_proxy_server` (which is whatever was set before geo-fix) — this is actually correct behavior since `unset_wininet_proxy` restores the original value, not the geo-fix value. So `proxy_port` in state is only needed if something in cleanup needs to know which port was used. Given current cleanup logic, it's optional — but add it for debugging and future use.

**Coordination with Task 4**: The watchdog must be started after CA install (step 11 above) and the state must be saved before starting the watchdog (so watchdog can load it on crash). This task defines the startup order including watchdog's position; Task 4 implements the watchdog itself. Coordinate on the exact function call placement.
